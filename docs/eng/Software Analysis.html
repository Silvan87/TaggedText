To have a simple, minimal and expressive syntax to describe mainly textual contents, but in the future also binary data (e.g. byte, 16 bit integer, 32 bit integer, signed or not...), and to describe presentation templates that if they are applied to content files, they can produce distributable files (e.g. html pages, ebooks, showcase websites or any textual type file, in the future also binary files).

1

As a content creator, I'd like a simple, minimal and expressive syntax to describe any textual content, so as to focus on the content and to have a source code of the content without thinking at all about the layout and graphics.

The syntax must be:

  • simple to read for a human, possibly understandable "at a glance".
  • minimal, i.e. use the minimum number of special characters and conventions while not limiting expressiveness.
  • comfortable, the main aid to the use of this language will be the highlighting of the text, except for this it must be possible to write as simple text without the need of complex tools such as development environments.
  • broadly expressive, the html language can be used as a reference for expressiveness. It must be possible to define a textual content by qualifying the pieces of text with an adjective (or a tag). These chunks of text can be arbitrarily defined in number, length, start of substring and end of substring. These qualified (or tagged) pieces of text must be able to be inserted into a hierarchical structure with as many levels as the user needs.
  • focused on content and therefore flexible, with a view to having models for generating layouts and graphics, the syntax should semantically qualify the parts of the text so to use them with great freedom at a later time, one should not feel the need to think about the arrangement of the contents, at most just a sequential order of the same.

The syntax will remain open to improvement if any future expressiveness limits are inconvenient.

As a content creator, I would like to use the syntax defined with US-1 to describe templates that utilize content written with the same syntax to potentially generate any distributable textual file.

The syntax defined with US-1, called TT (Tagged Text), must be usable to describe special template files that enable to use texts in TT format, to apply rules through TT template files, and potentially to generate any textual file.

To simplify the technical work, both source files containing content and template files, as well as distributable files, will use UTF-8 encoding by default.

The distributable files produced, also referred to as publications, must be at least in HTML format, but there should be no specific restriction or binding to HTML format, so any textual format should be producible.

4

As a content creator, I would like to be able to write a minimal spine file with a few content files and a template file to make some publications with the main and basic TT mechanisms working

A command must be provided to use a spine file that will direct and write the publication.

The programming language to be used must be determined.

For the initial developments, where code refactoring may be frequent and have a significant impact, the E2E testing approach will be used. Since there is no UI for the TT libraries, the method called to fulfill the user's initial command will be the starting point for E2E tests, and the files produced by that method will be the endpoint to verify if what has been produced corresponds to the expected output.

Once the code organization becomes more mature, the Test-Driven Development method will also be adopted to cover approximately 80% of the developed methods or those methods for which testing input and output is most significant.

Every architectural pattern that can avoid code repetitions, facilitate function extensibility, and ensure long-term code maintenance must be used. Code and its documentation must progress hand in hand.

The expected working examples, in accordance with the produced documentation, should cover: normal text, first and second level tagged text, in-line tagged text, tags without content, file-opening and file-ending rules (with sub rules: text, space, new-line, from-next-tag, from-var, from-file), rules for a tag (with sub rules: content, text, space, new-line, from-subtag, from-next-tag, from-var, from-file).

The implementation of secondary features such as comments and the escape character, as well as particularly complex features like working with lists, is excluded.

Note: This user story should be broken down into multiple user stories. However, considering the existence of a partially functional previous codebase, the intention is to incorporate the previous work, fix its defects, test it, and document it.

As a content creator, I would like to be able to use the from-next-tag subrule so that it retrieves tags of the same level and fixes the defect where it retrieves a value from the next tag but ignores the levels

It is currently implemented to retrieve the value of the next tag with the given name that is found. However, according to the documentation, it should only search in the same level as the tag to which the rule applies, but currently tag levels are ignored.

As a content creator, I would like to be able to use the escape character in all its cases (start of line, end of line, before special characters) with the expected effect on the escaped characters

The escape character is used to interpret as normal characters, therefore part of the content, characters that would otherwise be removed or considered special. There are 3 cases to manage:

  • white characters at the beginning of a line;
  • white characters at the end of a line;
  • special characters in any point of the normal content.
