Avere una sintassi semplice, minimale ed espressiva per descrivere principalmente contenuti testuali, ma in futuro anche dati binari (es. byte, integer 16 bit, integer 32 bit, con o senza segno...), e per descrivere dei modelli di presentazione che applicati ai file di contenuto possano produrre file distribuibili (es. pagine html, libri elettronici, siti web vetrina o qualsiasi file di tipo testuale, in futuro anche file binari).

1

Come creatore di contenuti, vorrei una sintassi semplice, minimale ed espressiva per descrivere qualsiasi contenuto testuale, così da focalizzarmi sul contenuto ed avere un codice sorgente del contenuto senza pensare minimamente al layout e alla grafica.

La sintassi deve essere:

  • semplice da leggere per un umano, possibilmente comprensibile "a colpo d'occhio".
  • minimale, ovvero usare il numero minimo di caratteri speciali e convenzioni pur non limitando l'espressività.
  • comoda, il principale ausilio all'uso di questo linguaggio sarà la colorazione del testo, salvo questo si deve poter scrivere come testo semplice senza necessitare di complessi strumenti come gli ambienti di sviluppo.
  • ampiamente espressiva, si può tenere come riferimento di espressività il linguaggio html. Si deve poter definire un contenuto testuale qualificando i pezzi di testo con un aggettivo (o un tag). Questi pezzi di testo possono essere definiti arbitrariamente in numero, lunghezza, inizio della sotto stringa e fine della sotto stringa. Questi pezzi di testo qualificati (o taggati) si devono poter inserire in una struttura gerarchica con un numero di livelli grande quanto l'utente ha bisogno.
  • focalizzata sul contenuto e quindi flessibile, nell'ottica di avere modelli per generare layout, impaginazioni e grafica, la sintassi dovrebbe qualificare in senso semantico le parti di testo per usarle con grande libertà in un momento successivo, non si dovrebbe avvertire la necessità di pensare alla disposizione dei contenuti, al massimo giusto un ordine sequenziale degli stessi.

La sintassi rimarrà aperta a miglioramenti se eventuali limiti di espressività futuri risulteranno scomodi.

2

Come creatore di contenuti, vorrei usare la sintassi definita con la US-1 per descrivere dei template che usino i contenuti scritti con la medisima sintassi per produrre potenzialmente qualsiasi file testuale distribuibile

La sintassi definita con la US-1, chiamata TT (Tagged Text), si deve poter usare per descrivere dei file speciali di template che permettano di usare testi in TT, applicare delle regole tramite i file template TT e produrre potenzialmente qualsiasi file testuale.

Per semplificare il lavoro tecnico sia i file sorgenti con il contenuto, sia i file template che i file distribuibili, useranno di default la codifica UTF-8.

I file distribuibili prodotti, anche detti pubblicazione, devono poter essere almeno in formato html, ma non deve esserci nessuna restrizione o legame specifico con il formato html, quindi qualsiasi formato testuale deve poter essere prodotto.

4

Come creatore di contenuti, vorrei poter scrivere un file spina minimale con qualche file di contenuto e un file modello per eseguire qualche pubblicazione con i principali e basilari meccanismi di TT funzionanti

Si deve fornire un comando per usare un file spina che dirigerà e scriverà la pubblicazione.

Si deve stabilire il linguaggio di programmazione che sarà usato.

Per i primi sviluppi, per i quali il refactoring del codice potrebbe essere frequente e di ampio impatto, si userà l'approccio di testing E2E. Non essendoci una UI per le librerie TT, il metodo chiamato per soddisfare il comando iniziale dell'utente sarà il punto di partenza dei test E2E ed i file prodotti da quel metodo saranno il punto finale di cui occorre verificare se quanto è stato prodotto corrisponde a quanto atteso.

Una volta che l'organizzazione del codice sarà più matura, si adotterà anche il metodo Test-Driven Development per coprire circa l'80% dei metodi sviluppati o comunque quei metodi di cui è più significativo testare input ed output.

Si deve usare ogni pattern architetturale che possa evitare ripetizioni di codice, facilitare l'estendibilità delle funzioni e la manutenzione del codice nel lungo periodo. Il codice e la sua documentazione devono procedere di pari passo.

Gli esempi funzionanti attesi, in accordo con la documentazione prodotta, devono riguardare: normale testo, testo taggato di primo e secondo livello, testo taggato in linea, tag senza contenuto, regola file-opening e file-ending (con sotto regole: text, space, new-line, from-next-tag, from-var, from-file), regola per un tag (con sotto regole: content, text, space, new-line, from-subtag, from-next-tag, from-var, from-file).

Viene esclusa l'implementazione di caratteristiche secondarie come i commenti e il carattere di escape e caratteristiche particolarmente complesse come il lavoro con le liste.

N.B. Questa US andrebbe scomposta in molte US, ma esistendo una base di codice precedente parzialmente funzionante, si intende incorporare il lavoro precedente, correggerne i difetti, testarlo e documentarlo.

6

Come creatore di contenuti, vorrei poter usare il carattere di escape in tutte le sue casistiche (inizio riga, fine riga, prima di caratteri speciali) con l'effetto atteso sui caratteri coinvolti

Il carattere di escape (uscita) serve per interpretare come caratteri normali, dunque parte del contenuto, dei caratteri che altrimenti sarebbero rimossi o considerati speciali. Le casistiche da gestire sono 3:

  • caratteri bianchi ad inizio riga;
  • caratteri bianchi a fine riga;
  • caratteri speciali in qualsiasi punto del normale contenuto.

8

Come creatore di contenuti, vorrei poter ampliare la regola per il contenuto di una lista di tag e poter usare il sotto tag ##item più volte così da applicarlo rispettivamente al primo, al secondo e così via elemento, l'ultimo sub tag ##item disponibile verrà applicato ai rimanenti elementi

Attualmente il comportamento predefinito in caso il sotto tag ##item in una regola per una lista di tag sia usato più volte è che l'ultimo sotto tag ##item sovrascrive le istruzioni precedenti. Invece, per avere più flessibilità nella regola per una lista di tag, elencare più sotto tag ##item permette di applicare questi modelli ai rispettivi elementi: il primo sotto tag ##item al primo elemento (es. definire un'intestazione di una tabella), il secondo sotto tag ##item al secondo elemento e così via, l'ultimo sotto tag ##item si applica a tutti gli elementi rimanenti. Se ci sono più sotto tag ##item degli effettivi elementi da gestire, questo non deve essere un problema, ci si ferma quando gli elementi finiscono e non ha importanza se alcuni sotto tag ##item restano di fatto non utilizzati.

11

Come creatore di contenuti, vorrei poter personalizzare nel file spina anche la cartella dei file tt di contenuto

Nel file spina occorre introdurre il tag #content-path che definirà la cartella dei file tt di contenuto. Se il tag non è presente, allora la cartella sarà la stessa del file spina. Ciò richiede che la gestione della posizione del file spina e dei file contenuto possano essere diverse. Il test E2E dovrà personalizzare tutte le path attualmente personalizzabili (spine, file contenuto, file template, file pubblicazione).
